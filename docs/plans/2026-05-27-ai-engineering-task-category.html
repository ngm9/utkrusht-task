AI-Engineering Task Category (ADVANCED)

Draft 2026-05-27 Rev 2026-06-03 — gate & readiness Owner: Rohan

TL;DR. Add senior-IC competencies for engineers who’ve built and operated AI agents. Each task is build-it: the candidate gets a real agent codebase and builds / extends / hardens / fixes the relevant piece — the same task shape the pipeline already produces. The new parts are a python-ai-agent template (agent frameworks layered on the Python family) and the classifier routing to it from the competency. The gate is the existing persona eval critics + a run.sh readiness check (does the scaffold boot — exit 0/non-zero) — we drop the pytest pass/fail step: it fought the fix-it task shape (a “make the failing tests pass” task has no single blessed reference) and turned a non-deterministic real-LLM run into a coin-flip gate. The agent calls a real model, but only when the candidate runs it, on their own key — so the gate is now key-free: no API key sits in a sandbox at generation, and no per-gate-run LLM cost. Everything else — output schema, prompt keying, two-repo persistence — is reused unchanged. No new column, no new storage, no new task-shape machinery.

Platform prerequisite (cross-cutting, not AI-specific). This category depends on one platform change, documented separately: the Python template family + build-time inheritance (the substrate python-ai-agent is built on). Routing needs no change — the existing classifier picks python-ai-agent from its capability sheet once it’s registered; we are not feeding scope into the classifier. See Platform Prerequisites for the template-family detail. The rest of this doc covers only what is AI-engineering-specific.

What we’re trying to do

Build a way to assess engineers who have actually built and operated AI agent systems in production — not just watched the LangChain tutorial. That means senior-level competencies, tested the way you’d test a senior hire: hand them a realistic agent codebase and have them build, extend, harden, or fix the relevant piece with real frameworks. The scenarios we want to cover: experience building AI systems; multi-modal AI systems (e.g. Gemini over a document); agents in production; multi-agent systems in production; context engineering; tool use; and agent architecture (memory, orchestration, tool use).

What we have today

The pipeline. Competency + scenario JSON → the classifier picks a sandbox template → an LLM generates a task (description + starter code) → the gate runs the persona critics and a sandbox boot check → the task is stored to Supabase + GitHub. That machinery is solid and we reuse all of it.

The competencies. ~60 tech stacks across Beginner / Basic / Intermediate. A handful are AI-flavored — LangChain, LlamaIndex, RAG, Vector Databases, Prompt Engineering — but all at framework-tutorial depth (“wire a chain”, “build a basic RAG”).

The one task shape. Every task the pipeline makes today has the same form: here’s a repo with empty slots → write the code → we check it boots → ready/not-ready.

Things that already exist (so we don’t rebuild them):

  • The utkrusht-python template already bundles langchain llama-index openai anthropic + every web framework + DB clients, has Docker-in-Docker (tasks bring their own docker-compose.yml), tops up per-task requirements.txt at gate time, and even pre-bakes an embedding model (.fastembed_cache). It is the substrate the Python template family inherits from.
  • The E2B build flow (infra/e2b/templates/…/template.py + a declarative manifest.json capability sheet, manifest-hash landed in 45d398f) already supports composing one template from another at build time — the mechanism inheritance rides on.
  • The classifier (LLM-based, infra/classifier/llm_classifier.py) already maps a competency → a template by reading the template capability sheets, and routes the eval critic by persona. Used as-is — registering python-ai-agent (with its agent-framework capability sheet) + the new competencies is enough for it to route there; no classifier change.
  • The gate itself: the persona-routed eval critics (llm_task_eval + llm_code_eval) and the E2B sandbox check (infra/e2b/sandbox_eval.py — boot, docker compose up, pip install). We reuse these and have the sandbox check run run.sh for deployment-readiness (see the gate). We do not run pytest at the gate — see why below.
  • The output schema (ANSWER_CODE_SCHEMA), the prompt keying, and the two-repo persistence (template repo + answer repo) — reused unchanged.

The problem

  1. No senior-IC AI competencies. Our AI competencies test “can you use this tool,” not “have you shipped and operated an agent.” There’s no signal for someone who’s debugged a runaway agent at 3am or argued about a context budget.
  2. The candidate’s sandbox has no LLM access. An agent’s whole job is to call a model. So for the candidate to test their own work, the sandbox needs a working LLM endpoint during their session. Today there isn’t one. (Note: the gate does not need this — it never runs the agent; see the gate.)

How we’re solving it

Three moves, all inside the existing pipeline — no forked flow:

  1. Senior competencies (see below) — skill-areas like Microservices, not technologies like Django.
  2. A python-ai-agent template that inherits from the Python family — it layers ~6 agent frameworks (langgraph, pydantic-ai, crewai, mem0, langfuse, openinference) on the shared base; long-tail libs come per-task via requirements.txt. Inheritance keeps each image lean and isolates agent-framework pins from the web/data tasks. The candidate’s session carries a real LLM API key (routed through the existing Portkey gateway) so their agent code runs for real — see LLM access.
  3. One build-it task shape — give the candidate an agent codebase to build / extend / harden / fix. It’s the existing task shape with agent frameworks added; reuse ANSWER_CODE_SCHEMA and the existing prompt keying. The gate (persona eval critics + the run.sh readiness check) proves each task is realistic and deploys at generation time — unchanged from every other task, minus the pytest step.

The rest of this doc is the detail behind each move.

How a competency pins its framework

A competency is a skill-area, but a task has to be concrete: someone must pick the framework (the reference solution and the candidate self-check are written against one), and two candidates assessed on the same competency should face the same framework — otherwise their scores partly reflect which tool they happened to know, not the skill. A 25-minute task in an unfamiliar framework tests ramp-up speed, not architecture. So the competency does not stay open: it commits to a primary framework, disclosed to the candidate up front.

LayerOwns
Competencycommits to a primary framework (the canonical one for the skill) + a short list of explicitly-allowed alternates. The comparability anchor and the default.
Scenariodefaults to the competency’s primary; picks an allowed alternate only when we deliberately want cross-framework coverage — and the pick is disclosed in the brief via **Stack:**.
Templateinstalls the whole allowed set, so any allowed pick just runs.

The scope text still says framework-API trivia is not the bar — the skill is the architecture — but the framework is pinned and disclosed so the candidate isn’t ambushed by an unfamiliar API and scores stay comparable.

Task examples — build-it, not memo-it

Every task is a real codebase the candidate works inside. Candidate writes 100–300 lines using real agent frameworks. Grading is invariant-based — the candidate’s self-check and the human/persona reviewer look at the same invariants (loop terminates, tool called with valid args, cost ≤ ceiling, output parseable, quality ≥ baseline with a tolerance band), not exact strings. These invariants are not a generation-time pytest gate (see the gate) — they live in the candidate repo as a self-check and in the reviewer’s rubric.

CompetencyStack candidate usesWhat they build & how it’s graded
Production Agent Engineering LiteLLM + FastAPI + Anthropic/OpenAI SDKs + Langfuse Customer-support reply agent with model fallback, cost ceiling, streaming, and observability. Graded on: avg cost < $0.05/ticket, ≥8/10 eval cases correct, graceful fallback on 5K-token outlier.
Tool Use for Agents Anthropic tool use or OpenAI function calling, via LiteLLM Rewrite 23 broken tool definitions + add a RAG router for the tool catalogue. Graded on: wrong-tool rate < 10% on a 100-query eval set.
Multi-Agent Systems CrewAI or LangGraph Code-review crew (Planner→Reviewer→Critic→Tiebreaker), where Tiebreaker is NOT just another LLM call. Graded on: verdict accuracy, latency, cost, deterministic tiebreaker firing.
Context Engineering fastembed + tiktoken + Mem0 + Anthropic SDK Context-budgeting middleware + long-term memory around a provided agent. Graded on: p95 prompt size ≤ 12K tokens AND quality stays ≥ baseline on 30 Q&A pairs.

Every task is build-it — a real coding session, not a write-a-memo exercise.

Two worked examples (scenario → task)

To make “build-it” concrete, here are two scenarios as they’d appear in a scenarios file, plus a sketch of the task the pipeline generates from each — showing the invariant-style self-check, the candidate’s key at run time, and the readiness-only run.sh.

Example 1 — Production Agent Engineering

Scenario (free-form string in the scenarios file)

**Stack:** LangGraph + LiteLLM (Anthropic primary, OpenAI fallback)
**Domain:** customer-support reply agent for an e-commerce SaaS
**Candidate writes:** agent/policy.py (model fallback + cost ceiling); bound the loop in agent/graph.py
**Provided broken:** an unbounded graph loop; policy hooks stubbed (NotImplementedError);
                     the agent hard-crashes when the primary model 5xxes
**Invariants:** loop terminates in <= 6 model calls; on primary 5xx the agent falls back to the
                secondary model via the LiteLLM router (not a try/except swallow); projected
                per-ticket cost <= $0.05 enforced BEFORE the call; >= 8/10 canned tickets get a
                non-empty, on-policy reply
**Senior signal:** fallback via the router, cost ceiling pre-call, bounded loop

Generated task (ANSWER_CODE_SCHEMA → {files, steps})

agent/ graph.py # LangGraph graph — loop node UNBOUNDED (candidate bounds it) policy.py # STUB: enforce_budget() + choose_model() raise NotImplementedError tools.py prompts.py # working invariants/test_agent.py # candidate-facing invariant tests (NOT a gen-time gate) fixtures/tickets.jsonl # 10 tickets + allowed policy labels run.sh # readiness self-check (LLM-free at the gate; key-gated ping in session) requirements.txt # langgraph litellm anthropic openai pytest .env.example # ANTHROPIC_API_KEY= / OPENAI_API_KEY=

Candidate brief: “This support agent runs away (unbounded loop) and hard-crashes when the primary model 5xxes. Make it production-safe: bound the loop, add LiteLLM model fallback, enforce a per-ticket cost ceiling, keep replies on-policy. Put your API key in .env and run ./run.sh.”

Invariants (the candidate runs these themselves; the reviewer reads them too): test_loop_bounded (a model that always says ‘continue’ → ≤ 6 calls, no hang); test_fallback_on_5xx (primary raises 529 → the router uses the secondary — asserted via the router/call-log, not exact text); test_cost_ceiling (a projected >$0.05 call is blocked pre-call); test_quality (≥ 8/10 fixture tickets → non-empty reply with an allowed policy label). These run on the candidate’s key, in their session — never as a generation gate.

run.sh (deploy readiness): python -m agent --selfcheck — imports the graph + policy modules, dry-loads prompts/tools, checks the fixtures load — exit 0. It does not hard-require the key (the gate runs key-free, and a candidate can start reading before setting .env); a key-gated ping fires only when a key is present.

run.sh + readiness probe (runnable)

#!/usr/bin/env bash
# run.sh
set -euo pipefail
pip install -q -r requirements.txt
python -m agent --selfcheck
echo "ready"
# agent/__main__.py  — invoked as `python -m agent --selfcheck`
import sys, os, json
from pathlib import Path

def _ping_model() -> None:
    # key-gated: only reached when a key is set (candidate session, never the gate)
    import litellm
    litellm.completion(model="anthropic/claude-haiku-4-5",
                       messages=[{"role": "user", "content": "ping"}], max_tokens=1)

def selfcheck() -> int:
    # 1. scaffold imports — modules load even though enforce_budget()/choose_model() are stubbed
    import agent.graph, agent.policy        # import only — do NOT call the stubs
    from agent import prompts, tools        # dry-load prompts + tool defs

    # 2. fixtures present and well-formed
    rows = [json.loads(l) for l in Path("fixtures/tickets.jsonl").read_text().splitlines() if l.strip()]
    assert rows and all("ticket" in r for r in rows), "tickets.jsonl malformed"
    assert Path(".env.example").exists(), ".env.example missing"

    # 3. key-gated ping — ABSENT at the gate (no key) -> skipped, stays key-free;
    #    PRESENT in the candidate's session -> proves endpoint/auth/model wiring (a bad key fails here)
    if os.getenv("ANTHROPIC_API_KEY"):
        _ping_model()
    else:
        print("note: no ANTHROPIC_API_KEY — skipping model ping (set it in .env to run your agent)")

    print("selfcheck OK")
    return 0

if __name__ == "__main__":
    sys.exit(selfcheck() if "--selfcheck" in sys.argv else 1)

Note: readiness never runs the unbounded graph loop or calls the stubs, and never hard-requires the key — so it passes at the gate (no key) and in a candidate’s session before they’ve set .env. The ping calls the model directly (not through the agent graph), so it can’t trip the very loop the candidate is meant to bound.

Example 2 — Tool Use for Agents

Scenario

**Stack:** Anthropic tool use, via LiteLLM
**Domain:** internal "ops assistant" with a catalogue of 23 tools (deploy, rollback, query-logs,
            page-oncall, ...)
**Candidate writes:** fix the 23 tool schemas in tools/schemas.py; implement tools/validate.py
                      (arg validation); implement tools/router.py (RAG router over the catalogue,
                      using the baked fastembed)
**Provided broken:** 23 tool dicts with planted bugs (missing required, wrong JSON-schema types,
                     name typos, free-text where enums belong); a router stub that dumps ALL 23
                     tools into context
**Invariants:** every schema validates against the tool spec; a missing required arg is rejected
                BEFORE dispatch; the router returns <= 5 tools and includes the gold tool for
                >= 90% of a 100-query eval set; end-to-end wrong-tool rate < 10%
**Senior signal:** validation at the boundary, RAG routing instead of all-tools-in-context
                   (context-cost discipline)

Generated task

tools/ schemas.py # 23 buggy tool defs (candidate fixes) validate.py # STUB: validate_args() raises NotImplementedError router.py # STUB: returns ALL tools (candidate makes it a fastembed RAG router) catalogue.md # human tool descriptions (router corpus) agent.py # working; calls the model with router + validate invariants/ test_schemas.py test_validation.py test_router.py # candidate-facing tests fixtures/eval_queries.jsonl # 100 queries -> expected tool name scripts/readiness.py run.sh requirements.txt (litellm anthropic fastembed pytest) .env.example

Candidate brief: “The ops assistant calls the wrong tool ~30% of the time and sometimes with malformed args; all 23 tools are shoved into every prompt. Fix the schemas, validate args at the boundary, and build a RAG router that returns only the relevant tools per query.”

Invariants: test_schemas_valid (all 23 validate; each has required; legal JSON-schema types); test_validation_rejects (a missing required arg raises before dispatch); test_router_topk (≤ 5 tools, gold tool present for ≥ 90% of the 100 queries); test_wrong_tool_rate (end-to-end, candidate’s key, wrong-tool rate < 10% with a tolerance band).

run.sh: import the tools/ modules, then check the router corpus + eval fixture load — exit 0. It deliberately does not validate the 23 schemas: fixing them is the candidate’s task, so the buggy schemas are treated exactly like a stub. LLM-free at the gate; key-gated ping in session.

run.sh + readiness probe (runnable)

#!/usr/bin/env bash
# run.sh
set -euo pipefail
pip install -q -r requirements.txt
python scripts/readiness.py
echo "ready"
# scripts/readiness.py
import os, json
from pathlib import Path

def _ping_model() -> None:
    import litellm
    litellm.completion(model="anthropic/claude-haiku-4-5",
                       messages=[{"role": "user", "content": "ping"}], max_tokens=1)

def main() -> int:
    # import only — validate_args()/route() are stubbed, and the 23 schemas are deliberately buggy
    import tools.schemas, tools.validate, tools.router
    from tools.schemas import TOOLS

    # do NOT validate TOOLS here: fixing the 23 planted schema bugs IS the task, exactly like a
    # stub. Readiness only proves the scaffold loads + the router corpus / eval fixture exist.
    assert len(TOOLS) == 23, f"expected 23 tool defs, got {len(TOOLS)}"
    assert Path("tools/catalogue.md").stat().st_size > 0, "empty router corpus"
    q = [json.loads(l) for l in Path("fixtures/eval_queries.jsonl").read_text().splitlines() if l.strip()]
    assert q and all("query" in r for r in q), "eval_queries.jsonl malformed"

    if os.getenv("ANTHROPIC_API_KEY"):   # skipped at the gate; fires in the candidate's session
        _ping_model()

    print(f"selfcheck OK — scaffold loads, {len(TOOLS)} tool defs, {len(q)} eval queries")
    return 0

if __name__ == "__main__":
    raise SystemExit(main())

Note: this is the sharpest case of the “don’t depend on the candidate’s unfilled work” rule — the “stub” here is buggy data (the 23 schemas), not a raising function. Validating it at readiness would fail the gate on the very thing the candidate is graded on fixing, so readiness loads the module but never validates the schemas.

The task shape: build-it

Every task gives the candidate a real agent codebase and asks them to build / extend / harden / fix the relevant piece. That is the same shape the pipeline already produces (a repo with incomplete or buggy code the candidate completes) — the agent-specific part is only that the repo uses agent frameworks and calls a real LLM through the candidate’s session key.

Because it is the existing shape, it reuses the existing machinery:

  • Output: the existing ANSWER_CODE_SCHEMA ({files, steps}) — an agent task is a code task with more files (agent code, candidate stubs, self-check, config).
  • Prompt selection: the existing (competency, proficiency) keying — one ai_agent_build prompt with the competency’s scope injected.
  • Routing: the classifier’s ResolvedPlan (template + persona) — unchanged.
  • Gate: the persona eval critics + the run.sh readiness check; infra boots from the task’s own docker-compose.yml.
  • Persistence: the existing template-repo + answer-repo + tasks row.

Pipeline walkthrough, top-to-bottom

1. CLI entry — unchanged

cli/main.py::generate_tasks. Same inputs (competency JSON, background JSON, scenarios JSON).

2. Classifier — routes to python-ai-agent as-is (no change)

The existing classifier routes the AI competencies to python-ai-agent with no code change. It’s LLM-based and reads the template capability sheets, so once python-ai-agent is registered (its sheet lists the agent frameworks) and the competencies exist, “Production Agent Engineering” matches it the same way every other competency matches its template. We are not feeding scope into the classifier — that was a considered enhancement we’re skipping.

Assumption: the classifier distinguishes python-ai-agent from the leaner python base by their capability sheets (the agent frameworks are the tell). If AI tasks ever mis-route to base python, that’s the point to revisit feeding scope.

3. Scenario load — read the directives

Scenarios stay free-form strings but gain load-bearing headers:

**Stack:** LangGraph + LiteLLM
**Domain:** customer support for e-commerce SaaS
**Candidate writes:** agent.py
**Invariants:** loop bounded, fallback on primary-model failure, cost < $0.05/ticket
**Senior signal:** model fallback via LiteLLM (not try/except), cost ceiling pre-LLM-call

**Stack:** fixes the concrete framework for this task; the rest is guidance for the gen LLM. There is no **Answer:** directive — every task is the one build-it shape. If **Stack:** is absent, the scenario defaults to the competency’s primary framework.

4. Prompt selection — one agent prompt × competency scope

Reuse today’s (competency, proficiency) keying. One ai_agent_build structure prompt, with the competency’s scope snippet injected for domain:

The LLM fills today’s ANSWER_CODE_SCHEMA ({files, steps}) — an agent task is a code task with more files (agent code + stubs + self-check + config). No new schema.

6. Gate — persona critics + run.sh readiness (no pytest)

No pytest step — see the gate for why, and readiness for what run.sh proves. The gate never runs the agent, so no LLM key is in the sandbox at generation.

Today: a failing task is fed back to the LLM with “fix it.” For agent tasks we surface the gate’s verdict kind — see Retry feedback.

The sandbox: the python-ai-agent template

The AI category runs in python-ai-agent, the agent-focused member of the Python template family (the family is split by framework family — python-base → python-web / python-data / python-ai-agent — not by datastore tier; each tier bundles the common DB clients). The family split, how build-time inheritance works, why the DB needs no schema change for it, and the open mechanism choice are all platform concerns — see Platform Prerequisites §2 and the classifier doc. This section covers only what python-ai-agent adds on top.

What python-ai-agent adds (AI-specific)

  1. ~6 agent frameworks on top of python-llm: langgraph pydantic-ai crewai mem0 langfuse openinference-instrumentation (langchain / llama-index / openai / anthropic are already in python-llm).
  2. Use the already-baked fastembed (from python-llm) for embeddings in retrieval / context tasks — do not add sentence-transformers (it drags in torch, ~2–3GB).
  3. A real LLM API key in the candidate’s session env (routed through Portkey), so agent code can call a model from inside the sandbox during their session — the one new piece of wiring. See LLM access below.
  4. Long-tail libs (a task that wants autogen/dspy/…) go in that task’s requirements.txt — topped up at gate time, as today.
The dependency-conflict question is scoped to this one image. crewai/langgraph/mem0 carry heavy, opinionated pins (pydantic-core, langchain-core, httpx). Because the family isolates them in python-ai-agent, the question is just “does python-ai-agent build green and pass its own readiness checks?” — the web/DB tasks on the leaner base are untouched. The dependency-conflict test is Phase 0 work.

TypeScript / Mastra?

Out of scope for now. Python covers 80%+ of agent work; a parallel TS template doubles maintenance for marginal coverage. Revisit when usage justifies it.

LLM access — a real model, on the candidate’s key

An agent’s whole job is to call a model, so its code has to reach an LLM to run at all. We use a real model through the existing Portkey gateway — the same path the pipeline already uses to call OpenAI/Anthropic. No mock, no fixtures. But the gate never runs the agent (we dropped pytest), so the only place a model is called is the candidate’s own session:

  • At generation, we do not run the agent. The gate is the persona critics (in the pipeline process, calling Portkey directly as today) + the run.sh readiness check (LLM-free). So no API key sits inside a sandbox at generation, and there is no per-gate-run LLM cost — a real simplification over the earlier “run pytest with our key in the sandbox” design.
  • During the candidate’s session, the candidate runs it, so it uses their key (set in .env / the session env; the SDKs read it — no code change in the repo). Per-candidate API spend stays off us, and the optional key-gated readiness ping confirms their endpoint works before they sink time in.

Because the model is real, grading is on invariants — the loop terminates within N steps, a tool was called with valid args, cost ≤ ceiling, the output is parseable, quality ≥ baseline with a tolerance band — not exact strings. These invariants ship as the candidate’s self-check and the reviewer’s rubric, not as a generation gate.

The gate — realistic + deployable, at generation time

An agent task runs the same gate every task does, simplified to two questions: is the task realistic, well-scoped, and solvable? (the persona critics) and does the environment boot? (run.sh’s exit code). Any failure retries with feedback.

StepWhat it does — and why it’s its own thing
1. Persona eval critics
generators/task/evaluator.py		 llm_task_eval + llm_code_eval — is the task realistic, well-scoped, the right difficulty, and is the senior signal actually present. Run in the pipeline; nothing in the sandbox. With pytest gone, these critics are also the solvability backstop, so llm_task_eval gets an explicit “could a senior finish this in ~25 min, and is there at least one clearly-good solution path?” check.
2. run.sh — readiness
the gate at gen time; sandbox_manager.py at deploy		 Brings the whole environment up — pip install, docker compose up the services, whatever the task needs — then a readiness probe, and exits 0 (ready) / non-zero (broken). This is the candidate-facing boot script; pip install lives inside it. It is LLM-free at the gate (see readiness).

Why we dropped the pytest gate

The original design ran the hidden tests against the reference solution at generation time to prove solvability. We remove that step, for two reasons that compound:

  1. A pass/fail gate fights the fix-it task shape. Our strongest senior tasks are “harden / fix this broken agent,” and the most realistic versions have no single canonical reference — many valid fixes exist. Encoding “the reference must make these exact tests pass” quietly narrows every fix-it task to one blessed solution and throws away the open-endedness that makes it a senior signal. (Sometimes “done” literally means a previously-failing test now passes — so a green-tests gate is the wrong contract.)
  2. Real-LLM + pass/fail is a category error. Invariant tests against a non-deterministic model are probabilistic (≥8/10, <10% wrong-tool rate). Gating generation on a probabilistic suite means a good task is discarded on unlucky sampling and a bad one slips through on lucky sampling — noise in the gate, not signal.

The invariant tests don’t disappear — they ship as a candidate-facing self-check (runnable by the candidate for their own feedback) and as a rubric input the human/persona reviewer reads. The senior signal lives in the rubric; the boot check lives in run.sh.

Readiness & run.sh — the deployability check

At generation the pipeline drops the generated candidate repo onto a fresh python-ai-agent E2B sandbox and runs run.sh wrapped in a wall-clock timeout, then reads the exit code. Exit 0 = “this task deploys cleanly, a candidate can start coding.” Non-zero = retry generation with the matching feedback label. run.sh is the deployability test — the same script the candidate runs and the same one sandbox_manager.py runs at deploy.

The rule that shapes every run.sh

Two constraints, easy to get wrong:

  • Don’t depend on the candidate’s unfilled work. run.sh runs against the candidate repo, where the stubs still raise NotImplementedError. So it may import modules and validate static artifacts (schemas, fixtures, configs, service connectivity) but must never call a stubbed function or run the agent loop. Importing a module whose function bodies raise is fine — the raise only fires on call.
  • LLM-free at the gate — not LLM-free always. The real constraint is “no our-key in the sandbox at generation,” not “no model call exists in the script.” So the readiness ping is key-gated: it fires only when a key is present.
python -m agent --selfcheck     # static checks always; model ping ONLY if a key is present

# end of selfcheck(), after the imports/fixtures/schema checks:
if os.getenv("ANTHROPIC_API_KEY"):   # absent at the gate -> skipped; present in the candidate's
    _ping_model()                    # session -> a 1-token call proves endpoint/auth/model wiring

At the gate there’s no key, so the ping is skipped — the gate stays key-free, deterministic, free. In the candidate’s session their key is set, so the ping fires and catches endpoint / auth / model-name misconfig the import check can’t — with the cost/flakiness landing on the candidate’s key, never on a gate signal. (“Does Portkey route to model X?” is a real question, but it’s a template-level check answered once when python-ai-agent is built — per-task readiness shouldn’t re-litigate a platform fact.)

What “ready” means for an agent task (the exit-0/1 signal)

For a web task run.sh ends by polling /health. An agent build task has no running service to poll, and its scaffold is deliberately incomplete, so “ready” means “the environment is set up and the scaffold loads, so the candidate can start coding”:

#!/usr/bin/env bash
set -euo pipefail                     # any failure -> non-zero exit -> "not ready"
pip install -q -r requirements.txt    # deps the scaffold + candidate need
docker compose up -d --wait           # vector DB / tool server, if the task has them
python -m agent --selfcheck           # imports the package, loads config / prompts / tool defs,
                                      # validates fixtures/schemas, pings services. Does NOT call
                                      # the stubs or the agent loop. Model ping only if key set.
echo "ready"                          # reached only if every step above exited 0

The readiness probe per task family:

Task familyReadiness probe (exit 0 = ready)
Web service (today)curl -sf localhost:8000/health returns 200.
Agent wrapped as a serviceSame — curl the agent’s /health / /invoke endpoint.
Agent as a CLI / looppython -m agent --selfcheck — imports + config + services load. No stub call.
Agent as a libraryA tiny script imports the entry point, asserts deps + config load, exits 0/1.
Task with a backing store (e.g. #4)docker compose up --wait then a dummy write/read/delete round-trip against the store.
Graph / orchestration (e.g. #6)build_graph().compile() — topology resolves without executing a node.

It stays LLM-free at the gate, so it needs no key there — the first real model call happens later, on the candidate’s key, once they’ve filled the stubs (or via the key-gated ping in their session). (run.sh shape is already persona-driven — e.g. persona="pm" ships no run.sh at all.)

Retry feedback — failure kinds

evaluator.py::build_retry_feedback already re-renders the gate’s verdict and re-feeds the failing candidate JSON so the LLM patches its own output. Because run.sh fails fast with its own exit code, the verdict keys off where it died. The test failed row is gone (no pytest gate); a selfcheck failed row replaces it:

Gate verdictFeedback to gen LLM
pip_failed“pip install -r requirements.txt failed — a dependency is wrong or conflicts. Fix the requirements.”
compose_failed“docker compose up failed — the task’s service containers don’t start. Fix the compose file.”
selfcheck_failed“run.sh’s --selfcheck exited non-zero — the scaffold doesn’t import, or a fixture/schema is malformed. Fix the scaffold so it boots (without filling the candidate stubs).”
timeout“run.sh exceeded the sandbox timeout — likely a hang in setup. Make the readiness probe terminate.”
task / code eval failedThe persona critic’s blocking_issues are passed back verbatim (existing behaviour).

Anti-goals

  • No separate flows/ai_agent/ folder. One build-it shape inside the existing pipeline.
  • No new task-shape machinery. Reuse ANSWER_CODE_SCHEMA, the existing prompt keying, and the two-repo persistence — no new output schema, no new column, no new storage.

  • No our-key in a sandbox. The gate never runs the agent; the only model calls happen in the candidate’s session on their key (plus the key-gated readiness ping).
  • No mock-LLM. When the agent does run (the candidate’s session), it calls a real model through the existing Portkey gateway. Any in-flight mock-LLM work (e.g. the agent_runtime/ worktree) is retired in favor of the candidate-key approach.
  • No fat monolith. The Python runtime is an inheritance family; python-ai-agent layers agent frameworks on the shared base.
  • No DB schema for template inheritance. Inheritance is build-time only; the stored capability sheet is already resolved.
  • No sentence-transformers/torch in the image. Use the baked fastembed.
  • No TypeScript / Mastra template for now. Python covers it.
