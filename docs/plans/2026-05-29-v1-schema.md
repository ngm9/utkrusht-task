**Status:** Proposal — single source of truth for the v1 (Utkrusht admin panel) data model.
**Tracks:** [docs/plans/2026-05-22-task-generator-production-readiness.md](2026-05-22-task-generator-production-readiness.md) — Threads A + B.

## Purpose

The production-readiness plan calls for several new tables (combo cache, template
registry, conversations, generation jobs, per-run scenarios). Some have landed,
others have not, and a few have been **superseded** by the unified
classifier+template schema that shipped on this branch.

This doc is the **one place to look** to answer: *"what tables does v1 need,
what's in each one, who reads/writes it, and what's its lifecycle?"*

It does **not** introduce schema beyond what the plan calls for — it just
makes the v1 surface concrete and self-consistent before we start B4 + B5.

## Existing tables — what's there, why, what changes

### `tasks` — the generated assessment

**Purpose.** One row per *generated, evaluated, gated, and persisted* task.
Source of truth for the candidate-facing artifact. Written by
`generators/task/creator.py`; read by the backend (Aptitude Tests) when
serving a session.

**Today's shape (existing in dev/prod):**

```
id                       uuid PRIMARY KEY      -- aliased as task_id in code
created_at               timestamptz
pre_requisites           text
answer                   text
criterias                jsonb
is_deployed              boolean               -- E2B deploy state
is_shared_infra_required boolean               -- Docker required?
task_blob                jsonb                 -- title, definitions, hints, resources, …
readme_content           jsonb
eval_info                jsonb                 -- LLM + gate verdicts
solutions                jsonb
task_type                text[]                -- ['BUILD'] | ['PR_REVIEW'] | ['DESIGN_REVIEW']
task_intent              jsonb                 -- per-task USE of the template (added 2026-05-28)
-- joined via task_competencies:
-- competencies …
```

**v1 change — B5 reorder.** Add a `status` column so create_task can
INSERT-then-build rather than build-then-INSERT:

```sql
ALTER TABLE tasks
    ADD COLUMN IF NOT EXISTS status text NOT NULL DEFAULT 'ready'
        CHECK (status IN ('draft','ready','failed','broken'));

CREATE INDEX IF NOT EXISTS tasks_status_idx ON tasks(status)
    WHERE status IN ('draft','failed','broken');
```

| `status` | Set by | Meaning |
|---|---|---|
| `draft`   | step 1 of new `create_task` | row inserted; GitHub repos + gist not built yet |
| `ready`   | last step of new `create_task` | repos + gist exist; candidate-servable |
| `failed`  | exception in steps 2–5 | mid-flight failure; partial artifacts cleaned up best-effort |
| `broken`  | reconciler (B5 follow-up) | drift found post-hoc (repo deleted, gist 404, etc.) |

**v2 hook.** Add `organization_id uuid NOT NULL DEFAULT '<utkrusht-uuid>'` when
opening to customers. Index `(organization_id, status, created_at DESC)`.

### `templates` — capability sheets for deployable environments

**Already shipped** via [migrations/2026-05-28-create-templates.sql](../../migrations/2026-05-28-create-templates.sql).

**Purpose.** One row per E2B template the classifier can match against.
Owns the build/test/install recipes. The unified-schema epic replaced the
legacy `template_registry`.

**Use case.** `runtime_resolver.resolve_plan(combo_key)` joins `task_template_match`
→ `templates` to get a `ResolvedPlan(match, template)` passed to
`infra/e2b/sandbox_eval.run_sandbox_eval` for the gate.

**No v1 change.** v2 may add `organization_id` only if customer-org templates
appear — not on the v1 roadmap.

---

### `task_template_match` — the combo cache (LLM match decision)

**Already shipped** via [migrations/2026-05-28-create-task-template-match.sql](../../migrations/2026-05-28-create-task-template-match.sql).

**Purpose.** Classify each `combo_key` *once*. `template_id IS NULL` is the
first-class "no built template matches" signal.

**Use case.** Cache lookup at the top of `runtime_resolver.resolve_plan`;
write on miss, read forever after. Invalidation keys: `classifier_model`,
`registry_version`.

**No v1 change.**

---

### `task_competencies` — junction (existing, unchanged)

**Purpose.** N:N between `tasks` and the upstream `competencies` table.
Written inside `create_task` after the `tasks` INSERT. No schema change for v1.

### Upstream / read-only — `competencies`, `backgrounds`, curated `task_scenarios_*` files

Out of scope for this doc — they live in the Aptitude Tests project (Supabase
`competencies` table) and as committed JSON under `data/generated/scenarios/`.
v1 keeps reading them as-is.

## New tables for v1

### `conversations` — the task-builder dialogue (B2)

**Why.** Today the chat that produces a `TaskBrief` lives in an in-memory
`SESSIONS: dict[str, SessionState]` in [task_builder/server.py:36](../../task_builder/server.py#L36).
Process restart = every in-flight builder session is lost with no trace.
The plan calls for persisting these so:

1. A user can resume a conversation after a refresh / restart.
2. The same conversation can produce multiple job runs (refine the brief and
   regenerate without re-doing the interview).
3. We have a record of *what the operator asked for* separate from *what the
   pipeline produced* (the job).

**Schema.**

```sql
CREATE TABLE IF NOT EXISTS conversations (
    id              uuid PRIMARY KEY DEFAULT gen_random_uuid(),
    -- v2: organization_id uuid NOT NULL
    -- v2: started_by      uuid       (user_id)

    messages        jsonb NOT NULL DEFAULT '[]'::jsonb,
                    -- [{role:'user'|'bot', content:str, ts:timestamptz}, …]
    brief           jsonb,
                    -- current best-known TaskBrief (snake_case TaskBrief model)
    missing_slots   text[] NOT NULL DEFAULT '{}'::text[],
                    -- what slots the bot is still asking for

    status          text NOT NULL DEFAULT 'active'
                    CHECK (status IN ('active','submitted','abandoned')),

    created_at      timestamptz NOT NULL DEFAULT now(),
    updated_at      timestamptz NOT NULL DEFAULT now(),
    submitted_at    timestamptz,
    abandoned_at    timestamptz
);

CREATE INDEX IF NOT EXISTS conversations_status_updated_idx
    ON conversations(status, updated_at DESC);
```

**State machine.**

```
active ──[POST /api/generate]──> submitted
   │
   └──[idle > 24h OR user closes]──> abandoned
```

**Use case.**

| Operation                              | Reads | Writes |
|----------------------------------------|-------|--------|
| `POST /api/session`                    |       | INSERT (`messages=[]`, `status='active'`) |
| `POST /api/chat`                       | row   | UPDATE `messages`, `brief`, `missing_slots`, `updated_at` |
| `POST /api/generate`                   | row   | UPDATE `status='submitted'`, `submitted_at=now()` + INSERT `generation_jobs` referencing this row |
| nightly sweep                          |       | UPDATE `status='abandoned'` WHERE `status='active' AND updated_at < now()-'24 hours'` |

**Relation to jobs.** One conversation → 0..N jobs. A "Generate again" with a
revised brief creates a *new* job pointing to the *same* conversation. The
conversation's `brief` is the *latest* version; each job's `brief` snapshot
is what was active at submit-time (immutable per job).

**v2 hook.** Add `organization_id`, `started_by`, and an RLS policy.
Backfill: `UPDATE conversations SET organization_id = '<utkrusht-uuid>'`.

### `generation_jobs` — the pipeline run (B3)

**Why.** Today a run is an in-memory `RUNS: dict[str, queue.Queue[StageEvent]]`
plus a worker thread in [task_builder/server.py:40-52](../../task_builder/server.py#L40-L52).
Process restart = the queue is gone, the thread is gone, the candidate gets
nothing. The plan calls for a durable jobs table + a Postgres-backed worker
loop so:

1. Restart-safe: a crash mid-run leaves a `running` row that can be claimed
   for retry or surfaced as failed.
2. Multi-worker: `SELECT … FOR UPDATE SKIP LOCKED` lets us scale to N workers
   without a separate queue infrastructure.
3. Observable: every run has a `log_url`, a `stage` cursor, timing fields.

**Schema.**

```sql
CREATE TABLE IF NOT EXISTS generation_jobs (
    id                  uuid PRIMARY KEY DEFAULT gen_random_uuid(),
    conversation_id     uuid REFERENCES conversations(id) ON DELETE SET NULL,
    -- v2: organization_id uuid NOT NULL
    -- v2: requested_by    uuid

    brief               jsonb NOT NULL,
                        -- snapshot of TaskBrief at submit time (immutable)

    status              text NOT NULL DEFAULT 'queued'
                        CHECK (status IN ('queued','running','done','failed','cancelled')),
    stage               text,
                        -- last stage label: '00_preflight' | '01_input_files' |
                        -- '02_scenarios' | '03_prompt' | '04_tasks' | 'done'

    log_url             text,
                        -- object-storage path for the per-run log bundle
    workspace_path      text,
                        -- /tmp/utkrusht-runs/<id>/ — set when worker starts
    result_task_id      uuid REFERENCES tasks(id) ON DELETE SET NULL,
                        -- populated on success
    error               text,
                        -- short error message on failure

    env                 text NOT NULL DEFAULT 'dev'
                        CHECK (env IN ('dev','prod')),

    -- Postgres-queue ergonomics
    locked_by           text,            -- worker hostname:pid
    locked_at           timestamptz,
    attempts            integer NOT NULL DEFAULT 0,
    max_attempts        integer NOT NULL DEFAULT 1,

    started_at          timestamptz,
    finished_at         timestamptz,
    created_at          timestamptz NOT NULL DEFAULT now(),
    updated_at          timestamptz NOT NULL DEFAULT now()
);

CREATE INDEX IF NOT EXISTS generation_jobs_claim_idx
    ON generation_jobs(status, created_at)
    WHERE status = 'queued';

CREATE INDEX IF NOT EXISTS generation_jobs_status_created_idx
    ON generation_jobs(status, created_at DESC);

CREATE INDEX IF NOT EXISTS generation_jobs_conversation_idx
    ON generation_jobs(conversation_id, created_at DESC);
```

**State machine.**

```
queued ──[worker claims via SKIP LOCKED]──> running ─┬─[success]─> done
                                                     │
                                                     ├─[exception]─> failed
                                                     │
                                                     └─[watchdog: locked_at < now()-'30m']─> queued (attempts++ until max_attempts; then failed)
```

**Use case.**

| Operation                                       | Reads | Writes |
|-------------------------------------------------|-------|--------|
| `POST /api/generate`                            | conversation | INSERT (`status='queued'`, `brief=snapshot`) |
| Worker claim loop                               | `WHERE status='queued' ORDER BY created_at FOR UPDATE SKIP LOCKED LIMIT 1` | UPDATE `status='running'`, `locked_by`, `locked_at`, `started_at`, `workspace_path` |
| Per-stage progress                              | row   | UPDATE `stage`, `updated_at` |
| Worker success                                  | row   | UPDATE `status='done'`, `result_task_id`, `finished_at`, `log_url` |
| Worker exception                                | row   | UPDATE `status='failed'`, `error`, `finished_at`, `log_url` |
| Watchdog (every 5 min)                          | rows where `status='running' AND locked_at < now()-'30 min'` | UPDATE → `queued` if `attempts<max_attempts` else `failed` |
| `GET /api/runs/{id}/events`                     | row + `log_url` tail | — (read-only stream) |

**Why Postgres-queue (not Redis/Celery).** Already calls Supabase from every
process; zero new infra; `SKIP LOCKED` gives multi-worker safety; the audit
trail *is* the data we need anyway. Concurrency capped by worker process count.

**v2 hook.** Add `organization_id`, `requested_by`; partial index
`(organization_id, status, created_at DESC)` for per-org dashboards.

### `generated_scenarios` — replaces the shared `task_scenarios.json` append (B4)

**Why.** Today [generators/scenarios/generator.py:219](../../generators/scenarios/generator.py#L219)
calls `save_generated_scenarios(..., append=True)` which **appends to a shared
file** like `data/generated/scenarios/task_scenarios.json`. Two concurrent
runs for the same proficiency level → JSON corruption (last writer wins, or
worse: two writers interleave bytes). It also leaves the repo dirty after
every run, violating the B4 invariant *"`git status` is clean after any
number of pipeline runs."*

**Two options the plan considers.** Per-run files in `/tmp` *or* a Supabase
table keyed by `(combo_key, proficiency)`.

**Decision: a table.** Per-run files solve concurrency but lose the *across-run*
benefit — today's scenarios JSON is also a **dedup memory** so we don't keep
asking the LLM for the same five scenarios on retry. The table keeps the
dedup memory and gets concurrency safety for free (Postgres upsert).

**Schema.**

```sql
CREATE TABLE IF NOT EXISTS generated_scenarios (
    id                uuid PRIMARY KEY DEFAULT gen_random_uuid(),

    combo_key         text NOT NULL,
                      -- canonical "Python (BASIC), Django (BASIC)" form;
                      -- same shape as task_template_match.combo_key

    proficiency       text NOT NULL
                      CHECK (proficiency IN ('BEGINNER','BASIC','INTERMEDIATE','ADVANCED','NO_CODE')),

    scenario_text     text NOT NULL,
                      -- one scenario per row (was one line in the JSON list)

    scenario_hash     text NOT NULL,
                      -- sha1(lower(trim(scenario_text))) for dedup

    domain            text,
                      -- optional domain tag if the LLM picked one
    focus_areas       text[] NOT NULL DEFAULT '{}'::text[],

    source            text NOT NULL DEFAULT 'scenario_generator'
                      CHECK (source IN ('scenario_generator','curated','operator')),

    generator_model   text,
                      -- e.g. 'gpt-4o-2024-08-06' — useful when we revise prompts

    -- v2: organization_id uuid NOT NULL

    created_at        timestamptz NOT NULL DEFAULT now(),

    UNIQUE (combo_key, proficiency, scenario_hash)
);

CREATE INDEX IF NOT EXISTS generated_scenarios_combo_prof_idx
    ON generated_scenarios(combo_key, proficiency, created_at DESC);
```

**Use case.**

| Operation                                                                  | Reads | Writes |
|----------------------------------------------------------------------------|-------|--------|
| `scenario_generator` start: load existing scenarios for `(combo, prof)`   | `WHERE combo_key=? AND proficiency=? ORDER BY created_at DESC LIMIT 50` | — |
| LLM returns N candidate scenarios                                          |       | upsert each via `INSERT … ON CONFLICT (combo_key, proficiency, scenario_hash) DO NOTHING` |
| `prompt_generator` / `create_task` reading scenarios for a combo           | same `WHERE` | — |
| curated JSON migration (one-off)                                           |       | bulk INSERT from existing `data/generated/scenarios/*.json` with `source='curated'` |

**Per-run scratch still needed.** A few stages genuinely want a local working
directory for files passed to the next stage (input bundles, intermediate
prompt text, gate logs). Per-run scratch goes to
`/tmp/utkrusht-runs/<job_id>/` and is *deleted at end* — see `generation_jobs.workspace_path`.
The repo's `data/generated/` tree exists only for tooling-friendliness and
holds gitignored stragglers.

**v2 hook.** Add `organization_id`; index becomes
`(organization_id, combo_key, proficiency, created_at DESC)`.

## Schema migration order

```
1. 2026-05-29-add-tasks-status-column.sql
     ADD COLUMN status; backfill all existing rows → 'ready'.
2. 2026-05-30-create-conversations.sql
3. 2026-05-30-create-generation-jobs.sql           (depends on conversations + tasks)
4. 2026-05-30-create-generated-scenarios.sql
5. 2026-05-30-backfill-generated-scenarios.sql     (load curated JSON; one-off)
```

Each migration is **additive**. None modifies an existing column's type or
constraint. Apply 1–4 on dev → smoke-test → apply on prod. (5) only ever
runs once per environment.

## Open decisions

1. **Per-conversation vs per-job log_url.**
   Recommend: per-job. The conversation isn't a stream — the *job* is.
2. **`workspace_path` cleanup.**
   Recommend: worker deletes the dir on `status IN ('done','failed')` after
   bundling the log_url. Watchdog rescues never-cleaned dirs older than 7d.
3. **`generated_scenarios.scenario_hash` algorithm.**
   Recommend: `sha1(lower(trim(scenario_text)))` — stable across machines,
   collision-resistant enough for this volume, indexable.
4. **Conversation message retention.**
   v1: keep forever. v2: org-configurable retention.
5. **`tasks.status='broken'` vs delete.**
   Recommend: never delete. Reconciler marks `broken` and stores the reason;
   manual review decides whether to regenerate.

## What this doc does NOT do

- It does *not* define the orchestrator/worker code shape — that's A2/B3
  implementation; see the production-readiness plan for the per-stage
  Generator+Verifier pattern.
- It does *not* re-spec `templates` / `task_template_match` — those are
  already canonical via [2026-05-27-unified-classifier-template-schema.md](2026-05-27-unified-classifier-template-schema.md).
- It does *not* introduce `organization_id` columns now — the plan is explicit
  that v1 is single-org and v2 adds the column + RLS as a data + policy change,
  not a schema rewrite. Every "v2 hook" line above marks where the column
  lands.
