Architecture & File Reference

Task Builder

A comprehensive guide to the task_builder/ package — what it is, how the pieces fit together, what every file does, the request/SSE data flow, and an element-by-element walkthrough of the web UI.

branch feat/task-builder 9 Python modules 3 static files FastAPI + SSE
For the design rationale behind the live stage-logs feature, see docs/specs/2026-05-19-task-builder-stage-logs-design.md.

1 · What it is

Task Builder is a local, single-user web app that replaces the command-line task-generation workflow with a conversation. Instead of hand-assembling competency / background / scenario JSON files and running four CLI tools in sequence, a hiring engineer:

  1. opens a browser at http://127.0.0.1:8000,
  2. chats with a bot that interviews them for the task inputs,
  3. clicks Generate task →,

and the app runs the full five-stage pipeline in a background thread, streaming each stage's live log output straight into the chat.

It is a thin conversational front-end + threaded wrapper around the existing run_pipeline.py linear orchestrator. It reuses that module's stage helpers but adds two things on top: an LLM interview that collects and validates the pipeline inputs, and live per-stage log streaming over Server-Sent Events (SSE).

PhaseDriverWhat happens
Interviewconversation.py The bot fills the TaskBrief slots, one topic at a time, validating each value.
Generationrunner.py The five-stage pipeline runs in a worker thread; logs stream to the browser.

Two in-memory registries in server.py hold all state:

  • SESSIONS — one SessionState per conversation (brief + chat history).
  • RUNS — one thread-safe queue.Queue per pipeline run; the worker thread pushes StageEvents onto it, the SSE endpoint drains it.

Both are plain dicts with no eviction and no persistence — they are lost on server restart, which is acceptable for a local single-user tool.

3.1 · __init__.py

Just the package marker / docstring. No exports.

3.2 · __main__.py

Makes the package runnable with python -m task_builder:

import uvicorn
if __name__ == "__main__":
    uvicorn.run("task_builder.server:app", host="127.0.0.1", port=8000, reload=False)
  • Binds to 127.0.0.1 only — the app is deliberately not reachable from other machines.
  • reload=False — it is a tool, not a dev server.

3.3 · server.py — the FastAPI app

The HTTP surface. Defines app = FastAPI(title="Task Builder"), mounts the static/ directory, and holds the two in-memory registries: 

SESSIONS: dict[str, SessionState]            # session_id -> conversation state
RUNS:     dict[str, queue.Queue[StageEvent]] # run_id     -> event queue
Method & pathPurpose
GET /Serves static/index.html.
GET /api/healthLiveness check — returns {"status":"ok"}.
POST /api/sessionStarts a conversation. Mints a session_id, stores a fresh SessionState, returns the bot's hard-coded greeting.
POST /api/chatRuns one conversation turn. Delegates to conversation.apply_turn. Returns reply, the full brief, missing_slots, ready.
POST /api/generateStarts a pipeline run for a session whose brief is_complete(). Mints a run_id, launches the worker thread, returns {run_id}.
GET /api/runs/{run_id}/eventsSSE stream of StageEvents for a run.

Key behaviours

  • _launch_run() creates a queue.Queue, registers it under the run_id, and starts a daemon thread running run_pipeline_for_brief, wiring the queue's .put as the emit callback. The pipeline runs entirely off the request thread.
  • chat() builds a fresh bot client and Supabase client per call.
  • get_supabase() is hard-wired to the dev environment — competency validation always hits dev Supabase, even when generation later targets prod (see §8).
  • The SSE generator runs the blocking queue.get() inside loop.run_in_executor(...) so the asyncio loop is never blocked. It JSON-encodes each event via dataclasses.asdict(event), and breaks the loop when event.stage == "done", then pops the run from RUNS.
  • generate() rejects an incomplete brief with HTTP 400; an unknown session/run is HTTP 404.

3.4 · conversation.py — the interview engine

The heart of the interview phase. The design is called Approach B: one LLM call per turn that returns structured JSON, rather than tool-calls or a multi-agent loop. Every bot response must be a single JSON object: 

{
  "reply": "<message shown to the user>",
  "slots_update": { "<slots learned this turn>": <value> },
  "ready_to_generate": true | false
}

Functions

  • build_bot_client() — an openai.OpenAI client pointed at the Portkey gateway with provider="anthropic". Model: BOT_MODEL = "claude-sonnet-4-6".
  • _extract_json(text) — pulls the first top-level JSON object out of an LLM response by brace-depth matching; tolerates surrounding prose.
  • _to_turn(parsed) — converts a parsed dict into a frozen ConversationTurn, defaulting every field.
  • run_turn(...) — runs one turn: build messages = system + history + user, call the LLM, parse. On unparseable JSON it retries once (the bad reply is inserted as an assistant message and a corrective user nudge is appended — roles must alternate, the Anthropic API rejects two consecutive user messages). If the retry also fails it degrades gracefully to a fallback reply instead of raising.
  • _clean_slots_update(...) — validates a raw slots_update before anything is stored. Returns (accepted_fields, rejection_messages). Proficiency → enum check; competencies → Supabase check with up to 5 "did you mean" suggestions; role/domain → trimmed strings; focus_areas → trimmed list; scenario_count → positive int only. Rejected fields are dropped.
  • apply_turn(...) — the full turn the /api/chat route calls: run_turn() → _clean_slots_update() → merge_brief() → a corrective turn if anything was rejected (rejection messages are appended to the user message as a SERVER VALIDATION: note so the re-ask reaches the user inside the same HTTP response) → append to history → return a ChatResult. ready = turn.ready_to_generate AND no missing slots — the server has the final say.

3.5 · slots.py — the brief data model

Defines the TaskBrief — the interview answers, owned authoritatively by the server (the LLM only proposes updates).

class TaskBrief(BaseModel):
    competencies:   list[str]  = []
    proficiency:    str | None = None
    role:           str | None = None
    focus_areas:    list[str]  = []
    domain:         str | None = None
    scenario_count: int        = 6
  • REQUIRED_SLOTS = competencies, proficiency, role, focus_areas, domain — five slots. scenario_count is not required (defaults to 6; the bot is told never to ask about it).
  • VALID_PROFICIENCIES = BEGINNER, BASIC, INTERMEDIATE, ADVANCED.
  • missing_slots() / is_complete() — completeness checks.
  • merge_brief(brief, update) — returns a new TaskBrief; never mutates (immutable update via model_copy).
Note on "six slots". Precisely, there are five interview slots the bot asks about, plus scenario_count, an auto-defaulted field the user never sees.

Also defined: Message (a role/content chat turn) and SessionState (session_id + brief + history).

3.6 · validation.py — slot validation

Ensures a slot value is real before it enters the brief.

  • SlotValidation — frozen result: ok, cleaned, error, suggestions.
  • validate_proficiency(value) — upper-cases and checks against the enum; on miss, the error lists the valid choices.
  • validate_competency(name, proficiency, supabase) — queries the Supabase competencies table for a row matching name (case-insensitive ilike) at the given proficiency. On a miss, uses difflib.get_close_matches (n=5, cutoff=0.5) to offer close spellings — the bot's "did you mean" suggestions.

3.7 · prompts.py — the bot's system prompt

A single SYSTEM_PROMPT string — the interview contract. It tells the LLM to:

  • collect the five slots and their shapes;
  • ask about missing slots one topic at a time, concisely and warmly;
  • never ask how many scenarios/tasks to generate, and never claim generation has started or finished;
  • set ready_to_generate: true only when all five slots are filled, and summarise + ask for confirmation first;
  • apologise and re-ask, offering suggestions, on an unknown competency;
  • obey the strict JSON-only output contract (no markdown, no fences).

3.8 · runner.py — the five-stage pipeline runner

Drives the generation phase. Executes the pipeline for a TaskBrief and calls an emit callback with a StageEvent before and after each stage.

StageEvent (frozen dataclass) — the unit streamed to the browser:

FieldMeaning
stageA STAGES label, or "done" for the terminal event.
statusrunning / ok / failed / completed, or log for a live chunk.
detailFree text — an error, an outcome, or a chunk of log output.
duration_sStage wall-clock time (on ok/failed).
outcomeStage-4 classification string.
task_id, task_urlSet on a successful done event.

STAGES = 00_preflight, 01_input_files, 02_scenarios, 03_prompt, 04_tasks.

run_pipeline_for_brief(brief, *, run_id, emit, env, runs_root)

  1. Picks the Python interpreter (_pick_python — prefers the project venv).
  2. Creates a per-run log dir: .task_agent_runs/web-<UTC-timestamp>-<run_id>/.
  3. Walks the five stages via the inner _stage() closure.
  4. Stops at the first failing stage and always emits a terminal "done" event (completed or failed).
  5. Any unexpected exception is caught and surfaced as a failed done event — a stage crash never kills the worker silently.

The _stage(label, cmd) closure is where live logs come from:

emit(running)
  → StageLogTailer.start()          # tails <label>.stdout + <label>.stderr
    → _run_stage()                  # blocks: runs the stage subprocess
  → StageLogTailer.stop()           # final flush of log files
emit(ok | failed)                   # with duration_s

The tailer's callback wraps each new chunk as StageEvent(stage=label, status="log", detail=chunk), so per-stage event order is always running → log… → ok/failed.

StageCommandNotes
00_preflighttask_agent_preflight.py --combo "<names>:<LEVEL>" --envSanity-checks the combo before expensive work.
01_input_filespython -m generate_input_files …Writes competency + background JSON.
(locate)_locate_input_files()Finds the JSON files stage 1 just produced.
02_scenariospython -m scenario_generator … --append + --focus-areas/--domainfocus_areas and domain from the brief are threaded in.
03_promptpython -m prompt_generator … --force --verboseSynthesises the tech-specific task prompt.
04_taskspython multiagent.py generate_tasks … --envThe actual task creation. --env must be threaded so a prod run stores in prod.

Outcome handling

  • _summarise_task_stage() reads stage-4 stdout and classifies it: TASK CREATED / EVAL GATE REJECTED / ERROR / UNKNOWN.
  • The done event is failed if the exit code is non-zero or the outcome contains REJECTED/ERROR/UNKNOWN — a task rejected by the eval gate is still a failure.
  • _extract_task_result() scrapes Task ID: and GitHub Repository: lines from stage-4 stdout to populate task_id/task_url on success.

3.9 · log_tail.py — live log tailing

StageLogTailer makes the per-stage logs appear while the stage runs, instead of only after it finishes.

  • Constructed with file paths, an emit callback, and a poll interval (interval_s, default 0.5 s).
  • start() spins up a daemon thread; stop() signals it and joins.
  • Each poll, _read_new(path) tracks a per-path byte offset, seeks, reads appended bytes, decodes utf-8 with errors="replace". A not-yet-created file yields an empty read.
  • The run loop uses Event.wait(interval) so stop() exits it promptly, and performs one final _drain() after stopping — so block-buffered output flushed at subprocess exit is never lost.
This module needed no changes to server.py — the tailer rides the existing SSE event queue by emitting ordinary StageEvents with status="log".

3.10–3.12 · static/ — the front-end

index.html — a minimal single page: a <header> (logo + title), <main> with the scrolling #chat area, and a fixed-bottom .dock (text input + Send button). No content is server-rendered; the page is an empty shell that app.js fills.

app.js — all browser-side logic. ~150 lines, no framework, no build step.

  • bubble(role, text, cls) — creates one chat row, returns the bubble element so callers can mutate it later. User text goes in via textContent (XSS-safe).
  • summaryCard(brief) — renders the task-brief card: a key/value grid of the five slots, an environment <select> (dev/ prod), and the Generate task → button.
  • startSession() — POST /api/session on page load.
  • send() — shows the user bubble + a … placeholder, POST /api/chat, replaces the placeholder with the reply; if data.ready, also renders the summary card.
  • startGeneration() — disables the controls, POST /api/generate, hands the run_id to streamRun.
  • stagePanel() — lazily creates one collapsible <details> panel per stage.
  • streamRun(runId) — opens an EventSource and maps each event to UI (see table below).
statusUI effect
runningopen the panel, summary ⏳ <stage>
logappend detail to the <pre>, auto-scroll
oksummary ✓ <stage> · <n>s, collapse
failedsummary ✗ <stage> <detail>, stay expanded
stage === "done"render the done bubble, close the stream

styles.css — a small, dependency-free light-theme stylesheet driven by CSS custom properties: chat bubbles, the .summary brief card, the collapsible .stage-log panels (custom ▸/▾ triangle, scrollable monospace <pre>), and the fixed input .dock.

8 · Known limitations & gotchas

State is in-memory and unbounded. SESSIONS and RUNS are plain dicts with no eviction and no persistence — everything is lost on restart. Fine for a local single-user tool; not safe for multi-user or long-running deployment.
Competency validation is always dev. get_supabase() is hard-wired to dev. A user can pick prod for task storage, but the competency they were validated against came from dev. If dev and prod competency tables diverge, a prod run can hit a competency the interview accepted.
  • Chat is synchronous. Each /api/chat call makes 1–2 blocking LLM calls on the request thread (a second one whenever a value is rejected).
  • The LLM proposes, the server disposes. The bot can set ready_to_generate, but apply_turn only reports ready when the brief is genuinely complete, and /api/generate independently re-checks brief.is_complete().
  • scenario_count is not interviewed. It defaults to 6; the prompt forbids the bot from asking about it.
  • Eval-gate rejections count as failures. A stage-4 task that is generated but rejected by the eval gate produces a failed done event, not a partial success.

10 · Relationship to run_pipeline.py

Task Builder does not re-implement the pipeline. runner.py imports the stage helpers directly from run_pipeline.py:

ImportedRole
REPO_ROOT, RUNS_DIR, SCENARIOS_FILEpaths
_pick_pythonchoose the interpreter for subprocess stages
_run_stagerun one stage as a subprocess, capture stdout/stderr/timing
_locate_input_filesfind the JSON files stage 1 produced
_summarise_task_stageclassify the stage-4 outcome

The difference is the stage loop: run_pipeline.py prints progress to a terminal; Task Builder's runner.py re-implements the loop to emit StageEvents and attach a StageLogTailer for live streaming. Everything below the stage loop — the subprocess execution, the log files under .task_agent_runs/, the outcome classification — is shared.

run_pipeline.py is a plain linear orchestrator: no coordinator agent, no per-stage verifier agents, no prompt-level escalation. The "smart" autonomous version is a separate Phase-2 design (docs/research/autonomous-task-agent/).
