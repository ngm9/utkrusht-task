Prompt Generator

DSPy agent that synthesises task-generation prompt templates for any (competencies, proficiency) combo. Scope-respecting, complexity-calibrated, category-correct. Ships uncompiled — empirical evidence shows few-shot demos hurt quality on this codebase.

→
TL;DR

Run python -m prompt_generator --name "Python, SQL" --proficiency BASIC. The agent gathers 5 context signals, asks one LLM to Generate a prompt file, asks a second LLM to Verify it against 4 hard-fail rules, then runs a deterministic Python Validator. Loops up to 5 times with feedback if anything fails. On pass, writes task_generation_prompts/<Level>/<tech>_<level>_prompt.py.

Decision: ship uncompiled. 4 compile experiments all regressed quality. Per-call signals already saturate the structural-pattern role that demos try to play.

01The decision: uncompiled wins

Three independent compile experiments + one blind-regeneration test, all returning negative lift:

ExperimentSetupLiftVerdict
Hand-built, gpt-4.1-mini, 4 demos$0-0.675Catastrophic
Sonnet 4 compile, 23 pairs, 3 demos, 2 holdouts~$1-0.035Noise
Sonnet 4 compile, 30 pairs, 3 demos, 8 holdouts~$1-0.021Noise
Blind regen, HTML+CSS, Sonnet 4 demos~$0.30-0.173Measurable drift
!
Why compilation hurts here

Five per-call signals already give the LLM strong structural examples for every call. Demos add a redundant fourth source of structural pattern, on top of three sources that already match the target combo better. With weaker matching to the user's combo, demos drift across task_category boundaries (see §13).

05The 5 context signals

The agent feeds five context sources to the LLM. Three from Supabase, two from local files. Together they give the LLM enough material to write a scoped, calibrated prompt template.

01

competency_scopes

db_queries.py · Supabase

Authoritative skill bounds. ~2K chars per competency. Source of truth for what the candidate can and can't be asked. Verifier's #1 hard-fail check is against this.

02

reference_prompts

retriever.py · local files

2-5 existing prompt files via 5-level fallback ladder. Provides structural template — section ordering, JSON shape, README format. ~10-60K chars.

03

similar_tasks

db_queries.py · Supabase

5-7 production tasks matching this combo, is_enabled + eval_info.passed. Calibrates complexity, file count, question style.

04

detailed_skill_signal

input_files.py · local files

Bundled: questions_prompt (30 sub-skill bullets) + role_context (YoE + responsibilities) + 3 example scenarios. ~2-4K chars. Empty for brand-new combos.

05

task_category

classifier.py · pure rules

Enum from rules-based classifier. One of pure_code, db_only, script_and_db, app_and_db, frontend, llm_framework, vector_db, messaging, microservices, non_code. Hard constraint on file shape.

+1

feedback_from_previous_attempt

verifier + validator · loop only

Empty on first attempt. Populated with concrete issue list on retries. Generate's docstring forces every issue to be addressed in the new output.

i
Why two separate LLM calls

If the same LLM both writes AND critiques, it tends to confirm its own output. A separate VerifyPromptSignature with different instructions (focused on the 4 hard-fail conditions) acts as a real adversary, not an echo chamber.

09What DSPy gives us

DSPy turns LLM calls into typed functions with declarative signatures. Without it, the agent code would be ~800 lines of f-string templates and brittle JSON parsing.

DSPy featureWhat you'd write by hand without it
dspy.Signature class200-line f-string template per LLM call
dspy.ChainOfThoughtManual "think step by step" prompt + reasoning extraction
Typed output fieldsjson.loads + regex + retry on malformed
dspy.Module compositionInline procedural code with strings
configure_dspy()Provider-specific HTTP clients scattered everywhere
Compile frameworkRoll your own search algorithm

Example — the entire Generate call is declared like this in agent.py:

class GeneratePromptSignature(dspy.Signature):
    """[docstring — the rules. ~150 lines with hard constraints
    for scope, structure, category, proficiency boundaries.]"""

    competencies:          str = dspy.InputField(desc="…")
    proficiency:           str = dspy.InputField(desc="…")
    task_category:         str = dspy.InputField(desc="…")
    competency_scopes:     str = dspy.InputField(desc="…")
    reference_prompts:     str = dspy.InputField(desc="…")
    similar_tasks:         str = dspy.InputField(desc="…")
    detailed_skill_signal: str = dspy.InputField(desc="…")
    feedback_from_previous_attempt: str = dspy.InputField(desc="…")

    new_prompt_file:       str = dspy.OutputField(desc="…")

# in the agent:
self.generate = dspy.ChainOfThought(GeneratePromptSignature)
out = self.generate(competencies="Python (BASIC), SQL (BASIC)", ...)
text = out.new_prompt_file   # already typed and parsed

DSPy reads the signature and renders the LLM prompt for you, parses the response back into typed attributes, and adds a reasoning field automatically when ChainOfThought is applied. None of this is compilation-specific. Even with compilation disabled, DSPy is still paying for itself just on these features.

10Compilation — and why we stopped

Compilation is an offline search that picks the 3-4 "best" few-shot demos from your training pool and saves them to compiled/agent_bootstrap.json. At runtime, those demos get prepended to every LLM call. The promise: examples beat instructions, so the agent should produce more consistent output.

Why it didn't help

What demos are supposed to do
  • Teach structural conventions (variable names, registry key shape)
  • Teach output style (section ordering, README format)
  • Provide concrete (input → output) examples of the function
Why per-call signals beat them here
  • reference_prompts (per-call) already shows 3-5 actually-similar examples
  • competency_scopes already bounds skills authoritatively
  • Demos are fixed; references are per-combo
  • Fixed demos can't match an arbitrary user query's task_category

16Future work

✓
Compilation might become viable when
  • Trainset of quality-signal prompts grows past 50 (currently 30)
  • A specific quality regression appears that demos could target
  • You switch to a weaker runtime model (e.g. Haiku) where demos help more

Other improvements worth considering

  • Fix the 20 prompt files with unparseable filename errors. These have __pycache__-confused parents or BOM characters. Cleanup PR would grow the trainset further and improve the retriever's fallback coverage.
  • tasks.task_category column in Supabase. Today task_category is an in-memory enum. Promoting it to a real column would enable persona-routed eval critics and category-aware retrieval at any scale.
  • Cross-combo embedding retrieval. retriever.py is filename-based. An embedding index over prompt sources could find better references at levels 4-5 of the fallback ladder.
  • Empirical eval gate. Post-generation, spin up an E2B sandbox and try running the task. Catches generation-level bugs that evals.py LLM-judging misses.
